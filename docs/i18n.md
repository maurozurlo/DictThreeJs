# i18n Architecture

## Stack

- **i18next** + **react-i18next** (translation hooks)
- **i18next-http-backend** (loads JSON from `/public/locales/`)
- **i18next-browser-languagedetector** (persists language choice)
- Config lives in `src/i18n.ts`

## Locale file locations

```
public/locales/
  en/
    menu.json
    laws.json
    deals.json
    meet.json
    daily_events.json
  es/
    menu.json
    laws.json
    deals.json
    meet.json
    daily_events.json
```

Add a new namespace by creating matching files in every language folder, then registering it in the `ns` array in `src/i18n.ts`.

## Key convention

**Every key stored in a data object (asset file) must include the namespace as its first path segment.** This mirrors the JSON structure so that calling `t(item.key)` with the right namespace "just works".

### Example — Deals

```ts
// src/assets/deals.ts
{ text: 'deals.1.text', acceptText: 'deals.1.acceptText', ... }
```

```json
// public/locales/en/deals.json
{ "deals": { "1": { "text": "A spy offers you...", "acceptText": "You accepted..." } } }
```

```tsx
// component
const { t } = useTranslation('deals');
t(currentDeal.text) // → deals.json["deals"]["1"]["text"]
```

### Example — Daily Events

```ts
// src/assets/dailyEvents.ts
{ key: 'daily_events.military_coup_attempt', ... }
```

```json
// public/locales/en/daily_events.json
{ "daily_events": { "military_coup_attempt": "A coup attempt is narrowly avoided." } }
```

```tsx
const { t } = useTranslation('daily_events');
t(event.key) // → daily_events.json["daily_events"]["military_coup_attempt"]
```

### Example — Laws (id-based, no key stored)

Laws do not store a key field. Their label is derived from `law.id` at render time.

```json
// public/locales/en/laws.json
{ "laws": { "labels": { "0": "Increase military spending by 50%", ... } } }
```

```tsx
const { t } = useTranslation('laws');
t(`laws.labels.${law.id}`) // → laws.json["laws"]["labels"]["0"]
```

## Using translations in components

Always load the namespace you need:

```tsx
// Single namespace
const { t } = useTranslation('deals');

// Multiple namespaces — use aliased destructuring
const { t: menuT } = useTranslation('menu');
const { t } = useTranslation('deals');
```

`useTranslation()` with no argument uses `defaultNS` (`'menu'`).

## Using translations outside React (store / utils)

Import `i18n` directly for translations needed inside Zustand actions or plain functions:

```ts
import i18n from '../i18n';

i18n.t('laws.labels.5', { ns: 'laws' });
i18n.t('daily_events.inflation_spike', { ns: 'daily_events' });
```

Note: log entries are translated at the moment they are appended to the log and stored as plain strings. They will not retroactively update if the user switches language — this is accepted behaviour.

## Adding a new translatable game data type

1. **Remove the hardcoded string field** from the TypeScript type (e.g. `label: string`).
2. **Choose a key strategy**:
   - If entries have a stable `id`, derive the key at render time: `t(\`ns.labels.${item.id}\`)`.
   - If entries are referenced by key elsewhere, add a `key: string` field following the `"namespace.key_name"` convention.
3. **Add locale entries** under the namespace-prefixed path in every language file.
4. **Update the component** to call `t()` instead of reading the raw string field.
5. **Update any store log entries** that stringify the field to use `i18n.t()`.

## Adding a new UI string

1. Pick the appropriate namespace (`menu` for general UI, a dedicated namespace for a feature area).
2. Add the key under the namespace prefix in every language's JSON file simultaneously — never ship with a missing translation.
3. Call `t('key.path')` with the right `useTranslation` namespace in the component.

## Namespaces and their scope

| Namespace | Scope |
|---|---|
| `menu` | All general UI — tabs, buttons, action panel, navbar, budget labels, log panel |
| `laws` | Law proposal UI strings and law labels |
| `deals` | Deal card text, outcomes, UI labels |
| `meet` | Meeting tab UI and action outcome strings |
| `daily_events` | Daily event headlines shown in the newspaper |
| `endscreen` | End screen tier names, flavour text, section headers _(Phase 2)_ |
| `shop` | Shop item names and descriptions _(Phase 2)_ |

## Languages

| Code | Language |
|---|---|
| `en` | English (fallback) |
| `es` | Spanish |

To add a new language: create a folder under `public/locales/` with matching JSON files for every namespace, then add the language code to the `supportedLngs` array in `src/i18n.ts`.
